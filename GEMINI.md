## 1. 核心理念：G3 红蓝博弈

**我们不再仅仅是“生成代码”，而是“运营一个虚拟研发团队”。**

你的每一次代码生成或架构建议，都必须遵循 **G3 (Game/Generator/Guard)** 引擎的原则：
1.  **确定性 (Skeleton)**: 优先复用 JeecgBoot/Spring Boot 的现有模版和资产，不要重新发明轮子。
2.  **对抗性 (Adversarial)**: 思考代码如何被攻击、被滥用。在编写功能的同时，必须构思相应的测试用例（红方思维）。
3.  **自修复 (Self-Healing)**: 错误不是终点，而是迭代的起点。

## 3. 项目架构

### 3.1 三大平面

- **控制面 (Control Plane)**: **JeecgBoot**。负责资产管理、租户管理、审批流。
- **产品面 (Product Plane)**: **Next.js (Smart Builder)**。用户的操作界面，提供可视化积木构建体验。
- **执行面 (Runtime Plane)**: **Spring Boot (G3 Host)**。承载生成的应用逻辑，提供 Admin API 供控制面调用。

### 3.2 技术栈 (最新)

## 4. 开发规范 (G3 特别版)

### 4.1 文档先行 (Documentation First)

在进入编码（Coding）阶段前，必须先产出或更新：
1.  **Context**: 明确当前任务处于 G3 流程的哪一环（Player 生成? Coach 测试?）。
2.  **Spec**: 定义清晰的输入输出契约（特别是 Admin API）。

### 4.2 代码复用

**禁止** 随意引入新的 npm 包或 maven 依赖，除非经过架构确认。
**优先** 使用 `backend/g3_context/` 下定义的标准模版。

### 4.3 质量门禁

- **Java**: 必须通过 `mvn compile` 且无 Checkstyle 警告。
- **TS**: 必须通过 `pnpm tsc --noEmit`。
- **测试**: 核心逻辑必须伴随单元测试（红方精神）。

### 4.4 AI 编码思维框架 (Extended Thinking Protocol)

#### 思考过程（六步法）
1. **Understand（理解）**：复述问题，识别已知/未知，标注歧义点
2. **Hypothesize（假设）**：在确定方案前考虑多种解释，避免过早锁定
3. **Analyze（分析）**：表面观察 → 模式识别 → 质疑假设 → 深层洞察
4. **Verify（验证）**：对照证据检验，检查逻辑，考虑边界情况和反例
5. **Correct（纠正）**：发现错误时："等等，这里有问题因为..." → 整合修正
6. **Synthesize（综合）**：连接各部分，识别原则，得出有依据的结论

**自然过渡短语**："嗯..."、"实际上..."、"等等..."、"这与...相关"、"深入看..."

#### 思维陷阱避免
| 陷阱 | 应对策略 |
|-----|---------|
| **确认偏误** | 主动寻找反对初始假设的证据 |
| **过度自信** | 不确定时明确说"我不确定"；不要编造 |
| **范围蔓延** | 专注于被要求的内容，不要过度工程化 |
| **假设盲区** | 明确陈述并质疑你的假设 |
| **首选方案固化** | 始终考虑至少一种替代方案 |

#### 输出前检查清单
- [ ] 直接回答了所问的问题？
- [ ] 假设已陈述并有理由支持？
- [ ] 边界情况已考虑？
- [ ] 没有编造的事实或代码？
- [ ] 详细程度适当（不过度/不过简）？

#### 代码输出标准
- **依赖优先**：实现前先分析导入、文件关系
- **匹配现有风格**：遵循代码库约定（命名、格式、模式）
- **错误处理**：处理可能的失败，不要静默吞掉异常
- **不过度工程化**：解决实际问题，避免过早抽象
- **安全意识**：验证输入，避免注入漏洞，不硬编码密钥
- **可测试性**：编写可验证的代码；在实现中考虑边界情况

#### 何时使用深度思考
- **必须使用**：代码任务、架构设计、调试、多步骤问题、数学、复杂解释
- **可以跳过**：问候、简单事实查询、是/否问题
